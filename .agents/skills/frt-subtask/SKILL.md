# FlowRT 子任务编排

## Overview

子智能体质量取决于任务包质量。主编排者必须先把上下文、边界、验收标准和交付格式说清楚，再产出提示词；收尾时必须亲自审阅 diff、跑验证、合并、清理已合入 worktree。

本 skill 中的“分发”默认表示**生成可复制给外部子智能体的任务提示词**，由用户手动派发。
除非用户明确说“由你直接派发 / 启动 subagent”，主编排者不得调用 subagent 工具。

任务包是具体执行材料，不是抽象规划。为节省主编排者收尾成本，任务包宁可长，也不能
让子智能体猜上下文、边界、验收标准、验证方式或交付格式。主编排者写提示词时应假设
子智能体只会看到这一段任务包，看不到本线程其它讨论。

每个子任务提示词必须是**完整自包含的可复制文本**。分发文档可以在顶部给出总览和
执行拓扑，但不能要求用户把“通用头部”“公共约束”或“交付格式”手动拼到某个任务
提示词前面。用户复制任意一个任务段给外部子智能体时，该任务段必须已经包含完整
背景、基线、worktree、范围、禁止事项、验证、提交和交付格式。

See [subtask-prompt-template.md](references/subtask-prompt-template.md) for the reusable prompt template.

## When to Use

- 用户要求并行派发多个 FlowRT 子任务时
- 用户交付若干 worktree，需要主编排者验收和合并时
- 子智能体反复做偏、漏测、误改版本、误写文档或提交过大时
- 主编排者需要把任务拆成可独立验收的修复、重构、文档或 CI 切片时

不要在以下场景使用：

- 单人可在当前 turn 内直接完成的小改动
- 还没有明确目标和重大决策的设计探索
- 用户明确要求不要使用子智能体

3. 指定基线分支、预期提交粒度和中文 Conventional Commit 规则
4. 指定必须运行的验证命令和可接受的未覆盖项
5. 指定交付回报格式：改了什么、验证结果、风险、未做事项

## Process

1. **准备主开发分支**：如果分发文档要求基于 `dev/vX.Y.Z` 或其他主开发分支，主编排者必须先用明确 seed commit 创建或确认该分支存在，并记录 `git rev-parse <branch>`。不要把创建主开发分支外包给第一个子任务，也不要让子智能体猜测改用 `master`。后续批次分发前，应确认该分支存在并基于最新 HEAD；分支因前置任务合入而前进是正常状态。
2. **切任务**：每个子任务只覆盖一个语义目标。不要把无关代码、文档、CI 和打包揉进同一个子任务，除非它们是同一行为变更的必要闭环。
3. **画拓扑**：用批次表和依赖边描述执行顺序。只有写集分离、语义不依赖彼此结果的任务才能放在同一批次；下游任务必须等前置任务验收并合入主开发分支后再启动。
4. **写任务包**：使用引用模板，提供 FlowRT 背景、当前决策、近期相关提交、路径、约束、验收标准、测试命令和禁止事项。宁可多写清楚，不要为省字让子智能体猜。任务包产出给用户，由用户复制给外部子智能体。每个任务包必须自包含；不要写“先复制通用头部再复制本任务”。
   对代码任务，任务包还应列出建议先看的函数、类型、测试文件和错误路径，并说明哪些
   现有行为必须保持不变。对 CI、打包、release 任务，任务包要列出 job 名、脚本名、
   artifact 名、cache key 和 release gate 依赖关系。
5. **要求 worktree 隔离**：每个子任务必须使用指定 worktree 和分支名。子智能体不得直接合主开发分支，不得清理别人的 worktree。
6. **要求原子提交**：每个可独立验收的行为变更一个提交，提交信息使用中文 Conventional Commits，正文列出要点。
7. **要求证据交付**：子智能体必须报告实际命令、退出结果、关键失败和未覆盖项。不能用“应该通过”“本地没跑但看起来没问题”代替验证。
8. **验收 diff**：主编排者先看 `git status`、`git log`、`git diff --stat` 和关键文件 diff。不要相信子智能体的完成声明。
9. **做语义审查**：先对照长期决策审查行为，再看代码风格。发现旧 fallback、旧版本、概念混淆、半接入主线或“测试绿但有 panic/warn 输出”时，不能合入。
10. **运行验证**：按任务包和风险补充验证。失败时主编排者亲自修，不把同一个收尾问题继续外派。
11. **合并到主开发分支**：优先 cherry-pick 或手工移植保持原子提交。冲突必须保留长期决策，不盲接外派分支旧逻辑。
12. **清理债务**：确认已合入、验证通过、无用户要求保留后，删除对应 worktree 和临时分支。不要保留 stash 作为历史债务。
13. **回灌经验**：如果收尾时发现提示词漏写的约束，把它加回后续任务包或本 skill，避免下一批重复返工。

## Execution Order Topology

每次分发必须先给出执行拓扑。推荐同时给出两种视图：

- **批次表**：`Batch N`、任务列表、前置条件、启动门槛。
- **依赖边**：`T01 -> T03` 这种边列表，说明为什么下游依赖上游。

拓扑规则：

- 同批任务的写集应尽量不重叠；如果不可避免，必须说明冲突文件和合入顺序。
- 下游任务不能基于“子智能体声称完成”启动，必须等主编排者验收、修补并合入主开发分支。
- 若任务只做调查、不改代码，也要说明它是否阻塞实现任务。
- 最终集成任务必须放在最后一批，且只能在所有前置任务合入后启动。
- 主开发分支创建是 Batch 0 之前的主编排者 pre-flight，不是普通子任务；分支不存在时不得派发任何基于该分支的任务。
- 拓扑变化时，先更新拓扑，再更新对应提示词，避免任务包和执行顺序不一致。

## Lessons From Recent FlowRT Subtasks

这些问题已经在 T05/T06/T07 收尾中真实出现，后续任务包必须预防：

- **基线漂移**：子任务基于旧 `dev/v0.4.0` 时，会漏掉刚合入的修复。任务包必须写清基线分支、必须从最新主开发分支创建 worktree，并要求交付时报告 `git log <base>..HEAD`。
- **旧 fallback 回流**：旧版本兼容、临时 fallback、被否决依赖容易被重新带回。任务包必须列出“不得回退”的长期决策，并要求用 `rg` 搜索旧关键字或旧版本号。
- **半接入主线**：只实现 runtime primitive、不接 codegen/CLI/self-description 时，要明确这是任务范围内还是缺口。任务包必须写清“完成”的用户可见定义。
- **假绿测试**：测试退出码为 0 但输出后台 panic、warning、timeout 退化或 skipped 核心路径，不能算通过。提示词要要求子智能体贴出异常输出和处理结论。
- **跨语言语义漂移**：Rust/C++ 分别能跑不代表互通。涉及 ABI、frame、backend、Service、self-description 时，必须要求跨语言 smoke 或 conformance。
- **依赖环境不清**：backend 任务必须写明私有依赖前缀、锁定版本和 CMake/Cargo feature；缺依赖时要说明查找路径和完整错误。
- **验证只跑最窄路径**：子智能体容易只跑自己新增测试。任务包要给最小必跑命令和高风险补充命令，主编排者验收时再跑合并后集成验证。
- **格式化遗留脏工作区**：合并冲突后可能出现已验证但未提交的格式化 diff。合并后必须再跑 `git status --short --branch`，工作树不干净不得宣布完成。
- **只改当前语言**：FlowRT 的语义通常跨 Rust、C++、CLI、IR、生成物和文档。任务包必须说明哪些边界要一起改，哪些边界本任务明确不改。
- **环境假设过强**：子智能体容易假设系统已装依赖或默认走 cargo 目录。涉及打包、backend、CI、cross-build 时，必须写明 FlowRT 私有依赖、系统安装路径和不允许隐式联网拉取的要求。
- **通用头部丢失**：用户手动复制任务时容易漏掉公共约束。每个任务段必须把公共上下文、禁止事项、交付格式重复写全；顶部总览只供人类阅读，不能成为任务执行依赖。

## Prompt Packet Checklist

- 任务目标：一句话说明要修什么或新增什么
- 背景上下文：相关版本计划、设计决策、已知坑和近期变更
- 范围边界：允许改哪些目录，禁止改哪些目录
- Worktree：明确目录名、分支名、基线分支
- 主开发分支：主编排者已创建或确认存在，并记录 `git rev-parse <branch>`
- 实现要求：行为、API、文档、CHANGELOG、错误处理和兼容策略
- 验证要求：必须跑的命令、期望结果、不能跑时如何说明
- 提交要求：提交粒度、提交格式、正文内容
- 交付格式：摘要、文件列表、验证输出、风险、后续建议
- 派发边界：明确“本提示词由用户手动转交；主编排者不直接启动 subagent”
- 执行拓扑：批次、前置任务、可并行任务、禁止提前启动的下游任务
- 当前基线：主开发分支名、预期最近关键提交、必须避开的旧逻辑和旧版本号
- 完成定义：说明本任务到什么用户可见程度才算完成，哪些缺口必须主动报告
- 负向检查：列出必须 `rg` 搜索的 forbidden string、旧 fallback、旧依赖版本或概念混淆点
- 跨边界验证：涉及 Rust/C++、CLI/runtime、codegen/runtime、backend/wire 时必须有端到端验证
- 合并后要求：主编排者要在主开发分支复验并清理 worktree，子智能体不得代替合并结论
- 开工前自检：要求子智能体报告 base commit、dirty 状态和创建 worktree 的命令
- 停止条件：基线不是最新、依赖缺失、范围冲突或需求不一致时先报告，不要猜测推进
- 自包含性：复制任意一个任务段时，不需要再拼接文档顶部的公共头部、公共禁止事项或公共交付格式
- 具体落点：指出建议先看的函数、类型、测试文件、脚本、CI job 或文档段落
- 不变量：列出不能改变的用户可见行为、语义层边界、兼容路径和安全边界
- 成功证据：至少指定一个成功路径输出、文件、测试或 smoke 结果
- 失败证据：至少指定一个失败路径错误文案、诊断输出或负向测试
- 验收口径：说明哪些命中是合法历史说明，哪些命中代表回归

## Common Rationalizations

| 借口 | 现实 |
|------|------|
| "子任务很简单，提示词短点就行" | 省掉上下文会把成本转移到主编排者收尾 |
| "字太多会影响执行" | 具体任务派发不是抽象规划，边界和验收越具体返工越少 |
| "让子智能体自己读代码理解" | 可以要求读代码，但必须告诉它优先读哪里、不能改哪里 |
| "子智能体说测过了" | 必须看命令、退出码和 diff，不能信口头报告 |
| "测试通过但输出里有 panic/warn" | 这不是干净通过，必须定位或说明为什么是测试刻意行为 |
| "先全部合进来再修" | 大冲突会吞掉长期决策，应按切片逐个验收 |
| "worktree 先留着以后再说" | 已合入 worktree 不清理会变成状态债务 |
| "提交标题写了就够" | FlowRT 要求中文 Conventional Commit，重要提交必须有正文 |
| "这个缺口属于后续任务" | 可以，但必须在交付中明确写出，不能让主编排者审查时才发现 |
| "依赖缺失所以没验证" | 必须先查项目私有前缀、锁文件和 CI 路径，再报告完整阻塞原因 |
| "我只负责 Rust/C++ 其中一侧" | 只有任务包明确限定单侧时才成立，否则要检查跨语言语义是否同步 |
| "负向搜索没必要" | 被否决方案最容易以旧字样和旧 fallback 形式回流，必须搜索 |
| "公共头部已经写在文档前面了" | 用户复制单个任务时会漏掉；每个任务提示词必须自包含 |
| "第一个子任务会自己建主开发分支" | 主开发分支是编排前置条件，必须由主编排者先创建或确认 |

## Main Orchestrator Review Gates

主编排者验收每个 worktree 时按 gate 前进；任何 gate 失败都先修复再继续：

1. **状态 gate**：worktree 是否干净；提交是否原子；提交信息是否中文 Conventional Commit 且有正文。
2. **基线 gate**：任务基线分支是否存在且等于分发文档记录的主开发分支；若不一致，先停下修正编排状态。
3. **范围 gate**：diff 是否只改允许目录；是否误改版本号、发布段、设计草案、别的 worktree 或用户代码。
4. **语义 gate**：是否符合 FlowRT 长期决策；是否引入旧 fallback、旧依赖、临时 envelope、CLI 隐藏职责或概念混淆。
5. **主线路径 gate**：实现是否接入 parser/IR/validator/codegen/runtime/CLI 中该任务声明的主路径；若只做底层 primitive，是否已明确标为非完整。
6. **跨边界 gate**：Rust/C++、CLI/runtime、static self-description/live socket、backend wire 是否有对应验证。
7. **异常输出 gate**：测试输出中的 panic、warning、skip、timeout、fallback message 都要解释；不能只看退出码。
8. **合并 gate**：合入主开发分支后复跑关键验证；再检查 `git status`。
9. **清理 gate**：确认 feature branch 是主开发分支祖先后删除 worktree 和分支；用户要求保留的 worktree 除外。

## Quick Reference

| 操作 | 命令 |
|------|------|
| 列 worktree | `git worktree list --porcelain` |
| 创建主开发分支 | `git branch dev/vX.Y.Z <base-commit>` |
| 确认主开发分支 | `git rev-parse dev/vX.Y.Z` |
| 看子任务提交 | `git log --oneline <base>..<branch>` |
| 看差异概览 | `git diff --stat <base>...<branch>` |
| 合入单提交 | `git cherry-pick <commit>` |
| 删除已合入 worktree | `git worktree remove <path>` |
| 删除已合入分支 | `git branch -d <branch>` |
